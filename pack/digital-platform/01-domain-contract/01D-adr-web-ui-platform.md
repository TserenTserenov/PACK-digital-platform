# ADR-001: Интерфейсная платформа IWE для международной аудитории

> **Расположение**: PACK-digital-platform / 01-domain-contract (архитектура платформы)
>
> Этот документ — governance (решение + обоснование), часть domain contract платформы.

---

## Разговор с архитектором

### 1. Постановка проблемы

**Владелец:** Сейчас единственная точка входа в IWE — Telegram-бот (@aist_me_bot). Я хочу выйти на широкую аудиторию. Не только русскоязычную — на самых разных языках. Работать в VS Code и с Git массы не будут. Что ты предлагаешь как дополнение к Telegram? Может Discord, Discourse или что-то другое?

**Архитектор:** Давай сначала зафиксируем ключевой принцип.

> **Принцип трёх интерфейсов:** Интерфейс платформы должен быть удобным для трёх типов взаимодействия:
> 1. **Человек ↔ ИИ** — персональное обучение, ассистент, ЦД
> 2. **ИИ ↔ ИИ** — агенты обмениваются данными, оркестрация
> 3. **Человек ↔ Человек** — сообщество, обсуждения, peer learning

Теперь — ограничения:

1. **Аудитория — международная.** Telegram популярен в СНГ, Ближнем Востоке, части Азии. Но не доминирует в США, Японии, Корее, Китае, Латинской Америке. Привязка к одной платформе = потеря большей части мира.

2. **Аудитория — массовая и нетехническая.** VS Code + Git — это инструменты разработчика. Даже Discord — это гик-аудитория (геймеры, разработчики). Нужен интерфейс с нулевым порогом входа.

3. **IWE — это персональная среда, не чат-сообщество.** Discord хорош для сообществ, но IWE — это личный инструмент: цифровой двойник, маршрут обучения, знания, аналитика. Для этого нужен rich UI (таблицы, графики, формы, визуализации), а не текстовый чат с кнопками.

**Владелец:** А Discourse? Он open-source, у него есть AI-плагин, MCP-сервер. И сообщество — это важно для обучения.

**Архитектор:** Discourse — сильная платформа для одного из трёх интерфейсов: человек ↔ человек. Форум, trust levels, модерация, 12 лет развития — это 9 из 10.

Но для человек ↔ ИИ — слабо. AI-плагин Discourse рендерит ответы ИИ как обычные посты. Нет интерактивных виджетов, нет карточек с tool use результатами, нет дашбордов. Sandbox ограничен 2 секундами на tool и 20 HTTP-запросами — невозможно построить цепочку агентов или обращение к ЦД. Нет per-user AI-памяти: бот не помнит контекст между разговорами с конкретным учеником.

Для ИИ ↔ ИИ — тоже не тянет. Есть REST API и MCP-сервер, но rate limits жёсткие, нет event streaming, агенты общаются через посты на форуме — это overhead, а не протокол.

И главное — IWE это **персональная среда**, а Discourse — **коллективная**. Дашборд с маршрутом обучения, визуализация ЦД, интерактивные упражнения по ZP — это не помещается в парадигму форумного поста.

**Вывод:** Discourse — потенциальный **адаптер** в Multi-surface архитектуре (слой сообщества на Phase 3), но не каноническая поверхность. Не проходит порог ЭМОГССБ: 53/80 при пороге ≥64.

**Владелец:** А OpenAI обещали Apps SDK, но заявка с декабря висит без ответа.

**Архитектор:** Не блокируйся на чужом approval. У тебя уже есть всё, чтобы построить собственную поверхность. И это стратегически правильнее — ты контролируешь обе стороны контракта.

### 2. Какие варианты рассмотрели

Мы оценили пять вариантов по шкале ЭМОГССБ (Эволюционируемость, Масштабируемость, Обучаемость, Генеративность, Скорость, Современность, Безопасность) + международный охват — 8 характеристик. Порог: ≥64 из 80 (средняя ≥8).

| # | Вариант | Балл | Вердикт |
|---|---------|------|---------|
| 1 | Telegram Mini Apps (TMA) как единственная поверхность | 55/80 | Отклонён |
| 2 | Standalone Web App (Next.js + AI SDK v6) | 68/80 | Принят |
| 3 | MCP Apps как единственная поверхность | 50/80 | Отклонён |
| 4 | Multi-surface (Web-ядро + адаптеры) | 69/80 | Принят (выбран) |
| 5 | Discourse (open-source форум + AI-плагин) | 53/80 | Отклонён |

#### Вариант 1: Telegram Mini Apps (TMA) — 55/80 ❌

| Э | М | О | Г | С | С | Б | i18n | Итого |
|---|---|---|---|---|---|---|------|-------|
| 7 | 6 | 8 | 5 | 8 | 8 | 7 | 6 | **55/80** |

- **Э (7):** TMA SDK развивается, но UI ограничен viewport Telegram.
- **М (6):** ~900M пользователей TG, но слабое проникновение в США, Японию, Корею, Латинскую Америку.
- **О (8):** TMA SDK хорошо документирован, сообщество активное.
- **Г (5):** Не создаёт платформу — привязан к парадигме мессенджера.
- **С (8):** Нативное ощущение внутри TG, быстрый запуск.
- **С (8):** Актуальная технология, но нишевая.
- **Б (7):** Telegram берёт на себя auth, но данные проходят через их серверы. Контроль над PII частичный.
- **i18n (6):** TG популярен в СНГ + Ближний Восток, слабо в остальном мире.

**Вердикт:** Отклонён. Привязка к аудитории одного мессенджера (~15% интернет-пользователей). Для международного масштаба — тупик.

#### Вариант 2: Standalone Web App (Next.js + AI SDK v6) — 68/80 ✅

| Э | М | О | Г | С | С | Б | i18n | Итого |
|---|---|---|---|---|---|---|------|-------|
| 9 | 9 | 8 | 8 | 8 | 9 | 8 | 9 | **68/80** |

- **Э (9):** Полный контроль над стеком. Любое изменение UI — без ограничений платформы.
- **М (9):** CDN + edge functions + SSR. Vercel/Cloudflare — глобальное покрытие.
- **О (8):** Next.js — самый популярный React-фреймворк, обширная документация.
- **Г (8):** Полноценная платформа: дашборды, формы, визуализации, AI chat.
- **С (8):** SSR + streaming + edge — <3 сек на первую загрузку.
- **С (9):** Next.js 15 + AI SDK v6 + MCP — cutting edge стек.
- **Б (8):** Полный контроль над auth (NextAuth), данными (Neon), hosting (Vercel). CSP, CORS, rate limiting — свои. PII в своей БД.
- **i18n (9):** next-intl — зрелое решение, ICU MessageFormat, автоматический роутинг по локали.

**Вердикт:** Принят. Самодостаточный вариант с максимальным контролем и охватом. Входит как ядро в Вариант 4.

#### Вариант 3: MCP Apps — 50/80 ❌

| Э | М | О | Г | С | С | Б | i18n | Итого |
|---|---|---|---|---|---|---|------|-------|
| 5 | 5 | 5 | 7 | 7 | 9 | 6 | 6 | **50/80** |

- **Э (5):** Спецификация 4 недели от роду, API может радикально измениться.
- **М (5):** Требует платной подписки на Claude/ChatGPT ($20+/мес). Не для массовой аудитории.
- **О (5):** Документация минимальна, примеров мало, сообщество формируется.
- **Г (7):** MCP-протокол перспективен для ИИ↔ИИ, но UI-возможности ограничены.
- **С (7):** Быстро внутри MCP-клиента, но зависит от хостинговой платформы.
- **С (9):** Bleeding edge — самый новый подход к AI-интерфейсам.
- **Б (6):** Данные пользователей проходят через Anthropic/OpenAI. Нет контроля над auth-слоем. PII у третьей стороны.
- **i18n (6):** Зависит от MCP-клиента. Claude.ai / ChatGPT — англоязычные по умолчанию.

**Вердикт:** Отклонён как primary. Перспективный адаптер для Phase 3 (когда спецификация стабилизируется).

#### Вариант 4: Multi-surface (Web-ядро + адаптеры) — 69/80 ✅ ВЫБРАН

| Э | М | О | Г | С | С | Б | i18n | Итого |
|---|---|---|---|---|---|---|------|-------|
| 9 | 10 | 7 | 9 | 8 | 9 | 8 | 9 | **69/80** |

- **Э (9):** Адаптеры эволюционируют независимо. Новая поверхность = новый адаптер, ядро не меняется.
- **М (10):** Любая поверхность, любой масштаб. Web + TG + MCP + будущие каналы.
- **О (7):** Сложнее Варианта 2 — нужно понимать архитектуру адаптеров. Но Web-ядро — тот же Next.js.
- **Г (9):** Платформа по определению. Каждый адаптер — новый канал дистрибуции. Работает как шаблон экзокортекса.
- **С (8):** Ядро быстрое (= Вариант 2). Адаптеры добавляют минимальный overhead.
- **С (9):** Backend-for-Frontend (BFF) + AI SDK + MCP — современный архитектурный паттерн.
- **Б (8):** Core-данные всегда в своей БД (Neon). Auth контролируется ядром. Адаптеры — тонкие прокси, не хранят PII.
- **i18n (9):** Наследует от Web-ядра (next-intl). Адаптеры получают i18n через API.

**Вердикт:** Выбран. Максимальный балл. Включает Вариант 2 как Phase 0-1, остальные поверхности — Phase 2-3 адаптеры.

#### Вариант 5: Discourse (open-source форум + AI-плагин) — 53/80 ❌

Оценка по принципу трёх интерфейсов:

| Интерфейс | Оценка | Комментарий |
|-----------|--------|-------------|
| Человек ↔ ИИ | 7/10 | AI-плагин с персонами, RAG, стриминг. Но: sandbox 2 сек на tool, нет structured output, нет per-user AI-памяти |
| ИИ ↔ ИИ | 5/10 | REST API + MCP-сервер. Но: rate limits, нет event streaming, нет agent-to-agent протокола |
| Человек ↔ Человек | 9/10 | Ядро Discourse: 12 лет развития, trust levels, модерация, chat + forum |

| Э | М | О | Г | С | С | Б | i18n | Итого |
|---|---|---|---|---|---|---|------|-------|
| 6 | 8 | 7 | 5 | 7 | 6 | 7 | 7 | **53/80** |

- **Г (5):** Форум не создаёт платформу для AI-native фич. Ruby/Ember.js — не мейнстрим для AI-команд.
- **С (6):** Стек зрелый, но не cutting edge для AI-задач.
- **Б (7):** Open-source, возможен self-hosting, хороший трек-рекорд безопасности. Но стандартный хостинг = чужие серверы.

**Вердикт:** Отличная community-платформа, но слабый AI-интерфейс. Может быть **компонентом** multi-surface архитектуры (слой сообщества на Phase 3), но не primary-поверхностью IWE.

### 3. Выбранное решение: Multi-surface с Web-ядром

**Архитектор:** Мы берём Вариант 4, который включает Вариант 2 как первый шаг.

```
           ┌─────────────────────┐
           │    Core Backend      │  Существующий: Node.js + MCP + Neon DB
           │    + Агенты          │  Стратег, Экстрактор, Проводник...
           │    + API Layer       │  Единые сервисы для всех поверхностей
           └──────────┬──────────┘
                      │
     ┌────────────┬───┼───────┬──────────────┬──────────────┐
     ▼            ▼   ▼       ▼              ▼              ▼
  Web App      TG Bot      TMA         MCP Apps       Discourse
  (Next.js)    (есть)      (Ph.2)      (Ph.3)         (Ph.3)
  КАНОНИЧЕСКАЯ                                        СООБЩЕСТВО
  ПОВЕРХНОСТЬ                                         + ДОСКА АГЕНТОВ
```

**Принцип:** Web App — каноническая (primary) поверхность. Telegram-бот остаётся как есть. Новые поверхности (TMA, MCP Apps) добавляются как тонкие адаптеры к тому же бэкенду. Discourse — двойная роль: сообщество + наблюдаемая доска агентов (агенты работают на бэкенде, публикуют ключевые моменты в Discourse для видимости).

**Почему Web — primary?**
- Работает в любом браузере, в любой стране, на любом языке
- Ты контролируешь обе стороны (frontend + backend) → минимальная связанность
- i18n — решённая задача в Next.js (next-intl)
- PWA (Progressive Web App) = опыт, близкий к нативному приложению
- AI SDK v6 даёт нативную поддержку MCP → все твои Pack-серверы доступны
- Верификация и оплата — любой провайдер (Stripe, Lemon Squeezy, Stars через TMA)

### 4. Что это значит для пользователя

**Владелец:** Хорошо. А как это выглядит с точки зрения обычного человека?

**Архитектор:** Вот три сценария.

#### Сценарий A: Новый пользователь (международный, не из Telegram)

```
1. Находит systemsworld.club (или ссылку из поста / поисковика)
2. Регистрируется за 30 сек (email + пароль или OAuth: Google/GitHub)
3. Видит дашборд: "Добро пожаловать в IWE"
   ├── Мой маршрут обучения (визуальная карта)
   ├── Мои знания (Pack viewer — поиск, навигация)
   ├── Мой цифровой двойник (текущее состояние, цели, прогресс)
   ├── Чат с ассистентом (AI, подключённый к Pack через MCP)
   └── Настройки (язык, тема, подписка)
4. Выбирает язык интерфейса (EN, RU, ES, ZH...)
5. Начинает с бесплатного тира → прогрессирует → платная подписка
```

**Ключевое:** Ноль зависимости от Telegram. Работает как обычный веб-сервис.

#### Сценарий B: Существующий пользователь Telegram-бота

```
1. Получает сообщение в боте: "Теперь доступен веб-интерфейс!"
2. Нажимает ссылку → попадает на Web App
3. Авторизуется через Telegram OAuth (один клик)
4. Видит ВСЕ свои данные из бота (ЦД, заметки, прогресс)
5. Продолжает пользоваться ботом для быстрых действий,
   а Web App — для глубокой работы (дашборды, аналитика, длинные сессии)
```

**Ключевое:** Бот не исчезает. Он становится "быстрым каналом", а Web — "глубоким".

#### Сценарий C: Пользователь платного семинара

```
1. Покупает доступ к семинару (Stripe / локальная платёжная система)
2. Получает личный кабинет в Web App:
   ├── Материалы семинара (структурированные по Pack)
   ├── Упражнения с проверкой (ZP-тренировка)
   ├── Персональное руководство (сгенерировано из программы + Pack + ЦД)
   ├── Чат с ассистентом по материалам
   └── Прогресс и сертификат
3. После семинара — аккаунт остаётся, переходит на бесплатный тир
```

**Ключевое:** Web App закрывает потребность в LMS (Learning Management System).

### 5. Что придётся сделать

**Владелец:** Конкретно — что мне нужно делать и сколько это займёт?

**Архитектор:** Разбиваю на фазы. Каждая фаза — отдельный РП с артефактом.

#### Phase 0: Фундамент (2-3 недели, ~15-20h)

| Задача | Что делаем | Оценка |
|--------|-----------|--------|
| **Репо** | Создать downstream-instrument репо `IWE-web-app` (Next.js + AI SDK v6 + TypeScript) | 2h |
| **Auth** | NextAuth.js — email/password + Google OAuth + Telegram OAuth | 4h |
| **i18n** | next-intl — русский + английский (каркас для других языков) | 3h |
| **API Layer** | REST/tRPC — прослойка между Web App и существующим бэкендом (Neon DB) | 4h |
| **Landing** | Главная страница systemsworld.club → Web App | 3h |
| **Deploy** | Vercel (frontend) + существующий Railway (backend) | 2h |
| **Артефакт:** | Работающий скелет: регистрация → пустой дашборд на двух языках |

#### Phase 1: Ядро (3-4 недели, ~25-30h)

| Задача | Что делаем | Оценка |
|--------|-----------|--------|
| **Дашборд** | Главная страница после входа: виджеты прогресса, быстрые действия | 6h |
| **Chat UI** | Чат с ассистентом (AI SDK v6 useChat + MCP tools) | 6h |
| **ЦД Viewer** | Визуализация цифрового двойника (данные из Neon) | 6h |
| **Pack Viewer** | Навигация по знаниям (подключение к knowledge-mcp) | 6h |
| **PWA** | Service worker, manifest — опыт мобильного приложения | 3h |
| **Артефакт:** | MVP — полноценный веб-интерфейс IWE |

#### Phase 2: Монетизация + LMS (2-3 недели, ~15-20h)

| Задача | Что делаем | Оценка |
|--------|-----------|--------|
| **Платежи** | Stripe (международный) + возможно Telegram Stars (через TMA-адаптер) | 6h |
| **Тиры** | Free / Pro / Seminar — Progressive UI по подписке | 4h |
| **Семинары** | Личный кабинет участника: материалы, упражнения, прогресс | 6h |
| **ZP-тренировка** | Интерфейс решения задач по нулевым принципам (РП #57) | 4h |
| **Артефакт:** | Платный продукт — Web App с подписками и LMS |

#### Phase 3: Адаптеры + масштаб (по мере спроса)

| Задача | Что делаем | Оценка |
|--------|-----------|--------|
| **TMA-адаптер** | Telegram Mini App как lightweight-вход в Web App | 8h |
| **MCP Apps** | Pack/ЦД как MCP App (когда спецификация созреет) | 6h |
| **Discourse** | Форум сообщества — peer learning, обсуждения, AI-модерация | 10h |
| **Локализация** | Испанский, китайский, другие языки | 3h/язык |
| **Analytics** | Grafana/PostHog — поведение пользователей | 6h |

### 7. Риски и митигации

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Распыление (Web App + бот + контент + семинар) | Высокая | Phase 0 — минимальный скелет. Не делать всё сразу |
| Auth-сложность (несколько провайдеров) | Средняя | NextAuth.js — зрелая библиотека, покрывает все варианты |
| i18n-качество переводов | Средняя | Начать с EN+RU, добавлять языки по спросу |
